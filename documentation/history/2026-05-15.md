# 2026-05-15 — Per-note ignore flag

Added support for a frontmatter property that lets users opt a single note out of serialization. Manual commands honor the flag and show a `Notice` explaining why nothing was done.

## Source code

- `src/app/constants.ts`: added `IGNORE_FRONTMATTER_KEY = 'dataview_serializer_ignore'` (underscores chosen for Dataview field-access compatibility).
- `src/app/utils/is-ignored-by-frontmatter.fn.ts` (new): pure helper. Returns `true` when the configured frontmatter key is "truthy" (boolean `true`, non-zero number, any non-empty string except `false`/`no`/`n`/`0`/`off` — case-insensitive, trimmed; arrays/objects are truthy). Returns `false` for missing, `null`, `undefined`, or explicit falsy values.
- `src/app/utils/is-ignored-by-frontmatter.fn.spec.ts` (new): coverage of truthy/falsy semantics, whitespace trimming, casing, complex value shapes, and missing-key cases.
- `src/app/plugin.ts`:
    - Added `isFileIgnoredByFrontmatter(file)` (synchronous; reads `app.metadataCache.getFileCache(file)?.frontmatter`).
    - `shouldFileBeIgnored` now short-circuits on the helper AFTER the excalidraw/canvas/extension guards but BEFORE the `force` early-return, so manual "Scan and serialize" commands also honor the ignore flag.
    - `serialize-current-file-dataview-queries` command checks the helper before processing and shows a `Notice` quoting the property name (`IGNORE_FRONTMATTER_KEY`).
- `src/app/refresh-button-extension.ts`: now receives an `isFileIgnoredByFrontmatter` callback alongside `processFile`. Both block-query and inline-query refresh-button click handlers check it before calling `processFile` and show the same `Notice` on skip.

## Decisions

- Underscored key (`dataview_serializer_ignore`) instead of dashes, so it can be referenced from Dataview as `file.dataview_serializer_ignore`.
- Truthy-string semantics (not strict boolean) so users aren't surprised when they write `: yes`, `: 1`, `: ignore`, etc. The narrow falsy set (`false`/`no`/`n`/`0`/`off`/empty) covers the realistic opt-out cases.
- Ignore flag is honored even on `force=true` (manual commands and forced-folder updates). Rationale: the flag is an explicit user signal, not an automatic-update heuristic. If users want a per-note flag that only suppresses automatic updates, they can already get that with the existing `QueryToSerializeManual:` query syntax.
- The whole-vault "Scan and serialize" command does NOT show a per-file notice for ignored files — that would be noisy, and the behavior matches how configured "Folders to ignore" are skipped today.
- Conversion / removal / insertion commands are NOT subject to the flag: they manipulate marker syntax but do not execute Dataview, so they remain useful (e.g. preparing a note before un-ignoring it).
- Implemented as a hard-coded property key rather than a setting. Keeps the surface area small; can be turned into a setting later without a migration if requested.

## Notes for manual verification

The plugin runs in Obsidian's GUI and cannot be self-verified by an agent. Manual checks:

- Add `dataview_serializer_ignore: true` to a note's frontmatter, save, and confirm queries are NOT updated automatically on subsequent modifications.
- Run "Scan and serialize Dataview queries in current file" on the same note and confirm a `Notice` appears that explicitly names the `dataview_serializer_ignore` property; confirm the file content remains unchanged.
- Click a refresh button (both block and inline) on a query inside an ignored note and confirm the same `Notice` appears instead of the usual "Dataview query serialized" message.
- Change the value to `false` and confirm queries get processed again on the next modify.
- Run "Scan and serialize all Dataview queries" across a vault that contains ignored notes and confirm those notes are silently skipped (no error, no spurious update).
